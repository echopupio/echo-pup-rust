# CatEcho 性能优化路线图（后续迭代）

## 1. 背景与目标

当前版本已将识别准确率拉高（`large-v3 + beam_search + 热词 + 谐音纠错`），但整体体感速度偏慢。  
本路线图目标是在不明显牺牲准确率的前提下，把平均交互延迟压到可接受区间。

目标分层：

- `T0`（马上可感知）：主观“松键到出字”明显变快。
- `T1`（工程目标）：P50 < 1.0s，P95 < 1.8s（不含网络 LLM）。
- `T2`（长期目标）：接近实时模式（流式增量，首字 < 400ms）。

---

## 2. 当前瓶颈拆解（基于现有实现）

### 2.1 STT 解码耗时高（主要瓶颈）

默认是准确优先配置：

- `model_path = ggml-large-v3.bin`
- `decoding_strategy = beam_search`
- `beam_size = 5`

这套配置对谐音错误控制很好，但 CPU 推理成本明显上升。

### 2.2 LLM 整理是串行阻塞路径（可选）

当 `llm.enabled=true` 时，请求是同步阻塞，额外增加网络和模型响应延迟。  
若网络波动，会出现长尾延迟。

### 2.3 线程数静态配置

目前 `n_threads` 是固定值配置，未根据机器核数和负载动态调整。  
在低核/高核设备上都可能非最优。

### 2.4 端点检测策略会影响“体感延迟”

若启用 VAD 自动结束，`vad_silence_threshold_ms` 默认 1500ms。  
这会增加“说完到触发处理”的等待时间（准确性好，但感觉更慢）。

### 2.5 缺少性能观测基线

当前没有统一的端到端耗时拆分指标（录音结束→STT→LLM→输入），优化容易“拍脑袋”。

---

## 3. 优化策略（按优先级）

## P0：先建立可观测性 + 快速收益（1-2 天）

目标：先看清耗时分布，再做低风险提速。

- 增加阶段耗时日志：
  - `stt_ms`
  - `llm_ms`
  - `postprocess_ms`
  - `e2e_ms`（从 stop 到 type_text 完成）
- 输出 P50/P95 的离线统计脚本（从日志聚合）。
- 增加文档化“速度优先临时配置”：
  - `decoding_strategy=greedy`
  - `greedy_best_of=2`（或 1）
  - `llm.enabled=false`（若对速度敏感）

验收：

- 可以稳定看到每段语音的阶段耗时分解。
- 能在同一台机器上重复对比不同配置延迟。

## P1：做可切换性能档位（2-4 天）

目标：把“准确率 vs 速度”做成产品能力，而不是手工改配置。

新增推荐档位（建议）：

- `accurate`：`large-v3 + beam_search(5)`（当前默认）
- `balanced`：`large-v3-turbo + greedy(best_of=2)`
- `fast`：`medium + greedy(best_of=1)`

实现建议：

- 新增配置 `performance_profile`，启动时自动覆盖一组参数。
- 保留手动参数优先级（profile 只是默认模板）。

验收：

- 用户一键切换档位，不改多项细节参数。
- `balanced/fast` 在延迟上明显优于 `accurate`。

## P2：并行与异步化（3-5 天）

目标：减少串行阻塞，降低长尾。

- LLM 改写从“强阻塞”改为“可选异步模式”：
  - 速度优先场景可直接输出 STT + 规则纠错结果。
  - LLM 仅在显式需要时启用。
- 尝试 STT 与后处理线程并行化（保持输出顺序一致）。
- `n_threads` 支持自动模式（按 `available_parallelism()` 动态设定）。

验收：

- LLM 不再拖慢默认路径。
- 在多核机器上吞吐提升且不会明显影响稳定性。

## P3：流式识别（中长期）

目标：从“说完才出字”过渡到“边说边出字”。

- 引入 streaming/分段推理：
  - 先出草稿，再做最终修正。
- 端点检测与增量解码结合，降低首字延迟。

验收：

- 首字延迟明显下降（目标 < 400ms）。
- 最终文本质量不低于当前离线方案。

## 5. 建议的近期执行顺序

1. 先做 P0（观测 + 临时速度配置），1-2 天内可见收益。  
2. 再做 P1（档位能力），让团队可稳定复用。  
3. 接着做 P2（异步与并行），压长尾。  
4. 最后推进 P3（流式），作为版本级能力升级。

---

## 6. 风险与取舍

- 速度提升通常会牺牲部分同音词准确率，需要通过档位化管理预期。
- LLM 异步化会带来一致性设计问题（是否回写、何时生效）。
- 流式方案工程复杂度高，建议先 PoC 再大规模改主链路。
