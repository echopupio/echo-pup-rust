# R-014: aria2 风格高速并发模型下载

## 需求概述

**需求描述**: 将模型下载能力从“稳定优先的串行 / 固定分片并发”进一步演进为 aria2 风格的高速并发下载方案，重点优化 `large-v3` / `turbo` / `medium` 等大模型的首次下载体验，并保留现有可追踪、可降级、可恢复的稳定性保障。

**验收标准**:
- 在服务端支持 `Range` 的前提下，模型首次下载可自动启用多连接并发下载，明显缩短大文件下载耗时
- 下载中断后可基于分片状态恢复，而不是只能回退到整单串行续传
- 下载过程中的进度、失败原因、降级路径在 TUI / 状态栏弹窗中持续可见
- 当服务端不支持 `Range`、代理行为异常或并发分片失败时，可自动回退到现有稳定下载路径

**优先级**: 中

---

## 技术分析

### 当前实现现状

| 模块 | 现状 |
|------|------|
| `src/model_download.rs` | 已具备断点续传、自动重试、代理失败回退直连、固定分片并发下载 |
| `src/menu_core.rs` | 负责下载动作发起、日志与进度事件汇总 |
| `src/status_indicator.rs` | 在 macOS / Linux 状态栏中展示下载弹窗、进度与错误 |
| 下载存储 | 主要基于单一 `.part` 文件；并发场景额外使用临时分片目录并在结束后合并 |

### 当前能力与 aria2 风格目标之间的差距

1. **续传粒度仍偏粗**: 当前 `.part` 续传以整单为主，尚未形成分片级持久化状态
2. **调度策略静态**: 目前分片大小与并发度固定，缺少 aria2 风格的动态切片与尾段再分配
3. **写入路径非最终形态**: 并发场景先落分片再合并，尚未演进为目标文件随机写入
4. **状态模型不完整**: 尚无独立 piece map / session 文件用于恢复、校验和诊断
5. **完整性保障可增强**: 缺少分片级校验、最终文件一致性校验与更细的失败归因

## 目标能力

### 必须具备

- 支持大文件模型的多连接并发下载
- 支持基于分片状态的恢复，而不是仅依赖整单 `.part`
- 支持服务端不兼容时自动降级
- 支持 TUI / 状态栏统一展示下载进度、分片状态与降级原因

### 期望具备

- 动态切片：慢分片或尾段可再切给空闲 worker
- 自适应并发：根据文件大小、网络状态和失败情况调整 worker 数
- 下载会话持久化：保留 piece state，便于下次恢复

### 暂不纳入本期

- 多镜像 / 多源下载
- BitTorrent / Metalink / RPC 控制接口
- 独立下载守护进程

## 技术方案方向

### 方案 A: 目标文件随机写入 + piece map（推荐）

核心思路：

1. 初始化目标临时文件与 piece map
2. 调度器按 piece 分配 `Range`
3. worker 直接写入目标文件对应偏移
4. 成功后将 piece 标记为完成
5. 中断恢复时读取 piece map，仅重下未完成区间

优势：

- 更接近 aria2 的真实工作方式
- 避免分片文件合并成本
- 更适合后续扩展动态切片与会话恢复

风险：

- 需要处理并发随机写入的一致性
- 需要额外设计 piece state 存储格式

### 方案 B: 保留分片文件，但补齐分片级元数据

核心思路：

1. 继续使用 `parts/` 分片文件
2. 增加 manifest / session 文件记录每片状态
3. 恢复时按 manifest 继续并发下载

优势：

- 对现有实现改动较小
- 易于快速补齐分片级续传

风险：

- 文件合并成本保留
- 后续升级到动态切片仍会受制于分片文件管理

## 风险与注意事项

1. **服务端兼容性**: Hugging Face / CDN 节点对 `Range` 与多连接支持程度可能不完全一致
2. **代理环境干扰**: 某些代理或本地加速器可能导致多连接表现更差，需保留直连降级
3. **磁盘写入一致性**: 随机写入或多分片合并时需避免文件损坏
4. **恢复语义复杂化**: piece map 一旦设计不稳，恢复路径会比当前 `.part` 更脆弱
5. **可观察性要求提高**: 并发下载若缺少分片级日志，定位问题会比串行模式更困难
