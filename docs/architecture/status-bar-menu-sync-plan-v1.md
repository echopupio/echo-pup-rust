# EchoPup 状态栏菜单与 TUI 功能同步方案 v1

最后更新：2026-03-19

## 1. 目标与范围

目标：将 `echopup ui` 中现有菜单能力完整同步到 macOS 状态栏菜单，确保两端行为一致、配置一致、下载逻辑一致。

范围：
- 同步 TUI 当前 15 个菜单动作到状态栏点击菜单。
- 抽离共享业务逻辑，避免 TUI 与状态栏两套实现漂移。
- 升级状态栏与主进程通信为双向 IPC，支持动作执行与状态回传。
- 覆盖配置编辑、热键校验、模型下载进度与日志展示。

不在本次范围：
- Linux 菜单栏实现。
- 新增与 TUI 无关的新配置项或新业务能力。

## 2. 功能对齐清单（与 TUI 一一对应）

1. 切换 `llm.enabled`
2. 切换 `text_correction.enabled`
3. 切换 `audio.vad_enabled`
4. 编辑 `hotkey.key`（支持按键捕获，最多 3 键）
5. 编辑 `llm.provider`
6. 编辑 `llm.model`
7. 编辑 `llm.api_base`
8. 编辑 `llm.api_key_env`
9. 编辑 `whisper.model_path`
10. 下载模型 `large-v3`
11. 下载模型 `turbo`
12. 下载模型 `medium`
13. 刷新本地模型列表
14. 保存配置
15. 退出 UI（仅关闭菜单/指示器进程，不退出 `echopup run`）

## 3. 总体设计

### 3.1 单一业务内核

新增共享菜单业务模块（建议 `src/menu_core.rs` + `src/model_download.rs`）：
- 统一封装菜单动作执行。
- 统一热键校验逻辑（复用 `hotkey::validate_hotkey_config`）。
- 统一模型下载与进度事件（复用现有重试、续传、无进度超时机制）。

TUI 与状态栏都调用该共享模块，不再在两个入口分别实现业务动作。

### 3.2 状态栏通信模型

将当前“主进程 -> 状态栏子进程”的单向消息，升级为双向 IPC（建议 NDJSON）：
- 主进程 -> 状态栏：状态变化、菜单快照、下载进度/日志
- 状态栏 -> 主进程：动作请求（toggle/edit/download/save/refresh/quit_ui）

状态栏只负责交互与展示；业务动作由主进程执行并回传结果。

### 3.3 状态一致性策略

- 每次动作完成后，主进程下发最新 `MenuSnapshot`。
- 配置类动作先更新内存态，`保存配置` 时统一落盘。
- 下载动作通过事件流驱动状态栏进度项，确保与 TUI 一致。

## 4. 分阶段实施计划

### Phase A：抽离共享能力（重构）
- 提取菜单动作枚举、动作执行器、菜单快照结构。
- 将 `ui.rs` 接入共享执行器，保持现有行为不变。
- 提取下载核心逻辑到共享模块，保留现有重试与超时策略。

验收：
- `echopup ui` 菜单功能行为与当前一致。
- 现有下载稳定性能力不回退。

### Phase B：双向 IPC
- 扩展 `status_indicator` 子进程协议：支持请求与响应。
- 增加消息类型：`ActionRequest`、`ActionResult`、`MenuSnapshot`、`DownloadProgress`。
- 主进程增加 IPC 处理循环，串行动作执行并回传状态。

验收：
- 状态栏发起动作后，主进程可执行并正确回包。
- 异常动作有错误信息回传，不导致子进程退出。

### Phase C：状态栏菜单落地
- 点击状态栏图标弹出完整菜单。
- 勾选项显示当前启停态。
- 编辑项支持原生输入框；热键支持捕获并校验。
- 模型下载项支持触发与并发保护（同一时间仅一个下载）。

验收：
- 15 个菜单项均可从状态栏触发。
- 编辑和切换类动作即时反映在菜单展示。

### Phase D：下载进度与日志展示
- 菜单中展示当前下载状态（模型名、已下载/总量、百分比）。
- 展示最近若干条下载日志（与 TUI 语义一致）。
- 下载完成/失败后刷新本地模型列表。

验收：
- 进度可见、失败可见、重试可见。
- 本地模型列表与实际文件一致。

### Phase E：回归与发布前检查
- 与 TUI 做逐项对照验收（15 项）。
- 校验“改动未保存”与“保存后生效”的边界。
- 校验 `echopup start/stop/status/restart` 与 `echopup ui *` 不受影响。
- 增加自动化验收入口：`./scripts/run_acceptance.sh`（覆盖 Phase E 契约测试 + 全量单测冒烟）。

验收：
- 状态栏菜单能力达到 TUI 完整等价。
- 无明显行为回归。

## 5. 风险与控制

- 风险：双端共享状态导致竞态。
  - 控制：动作在主进程串行执行，状态栏只做展示。

- 风险：下载状态在界面与实际任务不同步。
  - 控制：下载仅由主进程维护，界面只消费事件流。

- 风险：两端后续再次分叉。
  - 控制：禁止在 `ui.rs` 与 `status_indicator.rs` 写重复业务逻辑，统一经 `menu_core`。

## 6. 里程碑定义

- M1：完成 Phase A+B（共享内核 + 双向 IPC）。
- M2：完成 Phase C（菜单交互可用）。
- M3：完成 Phase D+E（可视化与回归完成）。
