# 业务需求文档（BRD）- echo-pup-rust

最后更新：2026-03-31

## 1. 业务背景

EchoPup 面向高频文字输入场景（写作、沟通、记录），核心价值是让用户通过热键语音输入（支持“长按说话”与“按压切换”）替代键盘输入。

当前业务痛点：
- 后台运行后，用户对当前录音/识别状态感知不足。
- 配置、模型下载、热键调整依赖 TUI，后台管理成本偏高。
- 文档历史路径与当前实现曾出现偏差，影响协作效率。
- 现有 Whisper 路径在中文口述场景下首字延迟和整句等待感明显，不适合作为长期实时输入方案。

## 2. 目标与价值

- 业务目标：
  - 降低后台运行场景的配置与维护成本。
  - 提升语音输入主链路稳定性与可感知性。
  - 将中文实时语音输入体验提升到“更快出字、更快 final、更低等待感”。
  - 建立可持续维护的文档治理基线。
- 用户价值：
  - 更低操作门槛：状态栏直接管理核心能力。
  - 更高确定性：录音、识别、下载状态更透明。
  - 更低等待感：中文短句说完后更快看到结果，并逐步具备 partial 草稿反馈。
  - 更低学习成本：需求、设计、实现与变更可追踪。
- 工程价值：
  - TUI 与状态栏共享业务逻辑，减少重复实现和回归风险。
  - 通过统一 ASR 抽象与 session 状态机，为后续模型替换和输入体验升级预留空间。

## 3. 范围边界

- 范围内：
  - 状态栏菜单与 TUI 功能对齐。
  - 热键安全校验与模型下载稳定性保障。
  - 文档体系标准化（版本化设计/技术、日期化变更日志、追踪矩阵）。
  - 将本地语音识别链路迁移为更适合中文实时输入的 `sherpa-onnx + SenseVoiceSmall` 架构。
- 范围外：
  - 与当前中文实时输入目标无关的新业务功能。
  - 首轮即实现所有宿主的复杂草稿替换或完整 IME preedit。
  - 以远程 STT 替代本地离线主链路。

## 4. 成功指标

| 指标 | 当前值 | 目标值 | 统计口径 |
| --- | --- | --- | --- |
| 状态栏菜单能力覆盖率 | 已覆盖部分 | 与 TUI 核心管理能力一致 | 菜单项逐项对照验收 |
| 语音链路延迟 | 已有埋点 | P50 < 1.0s, P95 < 1.8s（不含远程 LLM） | `e2e_ms` 日志聚合 |
| 首个 partial 延迟 | 当前未稳定达标 | P50 <= 350ms, P95 <= 700ms | `first_partial_ms` 日志聚合 |
| 句末 final 延迟 | 当前以停录后整段为主 | P50 <= 500ms, P95 <= 900ms | `final_after_silence_ms` 日志聚合 |
| 下载稳定性 | 已支持续传/重试 | 无进度超时可自动失败并重试 | 下载日志与失败复盘 |
| 文档追踪完整度 | 已建立基础结构 | 需求-设计-实现-验证可追踪 | 文档审计 + PR 检查 |

## 5. 风险与假设

- 关键假设：
  - 用户主要在 macOS 下使用状态栏进行后台管理。
  - 近期主链路迁移会优先在 Linux 场景验证，再逐步兼容 macOS。
  - 本地离线推理仍是核心默认路径，但识别后端将从 Whisper 逐步迁移。
- 主要风险：
  - 状态栏与 TUI 双入口状态不同步。
  - 双向 IPC 引入后，异常处理复杂度上升。
  - 文档若不同步代码变更，会再次产生事实漂移。
  - 流式 ASR、partial/final 状态机与宿主输入兼容性会提高实现复杂度。
