# ADR 0003 - 热键触发模式双态与状态栏自适应占位策略

日期：2026-03-19
状态：已采纳

## 背景

在状态栏交互增强后，用户反馈集中在两类问题：
1. 热键触发语义不够统一，存在“松开是否停止”的预期分歧与边界不稳定问题。
2. 状态栏图标在空闲态占位偏宽，激活态视觉反馈与紧凑布局之间难以兼顾。

同时，下载链路在部分代理环境下会快速失败，需要更稳健的回退策略与可追踪日志。

## 决策

1. 热键触发策略采用双模式配置化：
   - `hold_to_record`：长按 1 秒开始，松开结束；
   - `press_to_toggle`：长按 1 秒开始，下一次按下结束。
2. 触发链路使用显式状态机与按键序列控制（按下/释放序列号、当前按压是否已触发、停止防抖窗口），避免误停止。
3. 状态栏占位采用自适应长度：
   - 空闲态：窄宽度，仅展示 logo；
   - 激活态：加宽并展示边缘脉动胶囊，保持可视反馈。
4. 下载链路在代理失败时自动回退直连，并在失败后清理 0B `.part` 文件。

## 备选方案与取舍

1. 单一触发模式（仅长按或仅切换）  
   未选原因：无法覆盖不同用户习惯，且迁移成本高。

2. 继续沿用隐式按键状态判断  
   未选原因：边界抖动问题难定位，行为不可预测。

3. 固定状态栏宽度  
   未选原因：空闲态显得松散，激活态又容易压缩视觉元素，难同时满足。

## 影响

- 正向影响：
  - 热键行为可预测且可配置，减少误触发投诉。
  - 状态栏空闲态更紧凑，激活态反馈更明确。
  - 下载在代理异常环境下成功率提升，失败原因更可追踪。

- 负向影响：
  - 主流程状态机复杂度上升，需要保持测试覆盖。
  - 状态栏视觉参数（宽度、缩放、边距）需要持续校准。

## 后续约束

1. 任何热键触发语义变更都必须同步更新 `PRD`、`system-design`、`traceability` 与 `changelog`。
2. 状态栏空闲态默认不展示胶囊背景；激活态胶囊应优先保证可读与紧凑平衡。
3. 下载链路变更必须包含失败回退路径与日志可观测项。
